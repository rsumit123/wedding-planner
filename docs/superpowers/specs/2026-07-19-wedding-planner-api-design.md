# Sumit & Puja Wedding Planner API — Design

## Purpose

Provide a small persistent API for the Vercel wedding planner so organisers can manage tasks and guests together, while guests can RSVP without seeing private planning data.

## Public endpoints and hosts

- Frontend: `https://wedding.skdev.one`
- API: `https://wedding-api.skdev.one`
- Only `https://wedding.skdev.one` is an allowed browser origin for API calls.

## Access model

- Organisers use one shared account with username `sumit-puja`.
- The provided shared password is stored as a bcrypt hash in the VM-only environment file; it is never added to source control or returned by the API.
- Successful sign-in creates a 14-day secure, HTTP-only, `SameSite=Lax` session cookie scoped to `wedding-api.skdev.one`.
- All organiser write operations require a valid session.
- Guests do not sign in. They receive a public RSVP token that only grants access to their own invitation.

## Data model

### Events

Five seeded, immutable identity records are used in invitations and tasks:

1. Lagan & Tilak — 28 November 2026
2. Haldi & Matkor — 30 November 2026
3. Wedding ceremony — 1 December 2026, after 1:06 PM
4. Vidai — 2 December 2026, after 8 AM
5. Reception party — 3 December 2026

### Guests and invitations

- A guest represents an individual or household with a display name, optional phone number, and optional planning note.
- An invitation stores either `all_events=true`, or rows for explicitly selected event IDs.
- RSVP state is stored per invited event: `pending`, `accepted`, or `declined`.
- A guest with `all_events=true` sees each current event, including any future ceremony added by organisers.

### Tasks

Each task has a title, optional assignee name, optional deadline, optional event ID, status (`open` or `done`), and timestamps.

### Activity

Every organiser mutation writes an activity record with a concise action, timestamp, and actor username. Public RSVP submissions record a guest-facing RSVP activity without exposing contact details to public routes.

## API surface

### Authentication

- `POST /auth/login` — accepts username and password, sets a session cookie.
- `POST /auth/logout` — clears the session cookie.
- `GET /auth/me` — returns the organiser username or `401`.

### Organiser-only

- `GET /events`
- `GET`, `POST`, `PATCH /tasks`
- `GET`, `POST`, `PATCH /guests`
- `GET`, `POST`, `PATCH /invitations`
- `GET /activity`

### Guest RSVP

- `GET /rsvp/{token}` — returns only the named guest/household and their invited events.
- `POST /rsvp/{token}` — updates only RSVP states and an optional guest note for that token.

## Deployment architecture

```text
https://wedding.skdev.one      Vercel SPA
            |
            | HTTPS, credentialed CORS
            v
https://wedding-api.skdev.one  host Nginx and Certbot certificate
            |
            | http://127.0.0.1:8040
            v
wedding-api Docker container  FastAPI + SQLite at /app/data/wedding.db
```

- The container exposes `127.0.0.1:8040:8000`; no API port is publicly reachable.
- Nginx provides the standard proxy headers and redirects HTTP to HTTPS after Certbot has issued the certificate.
- Docker uses `restart: unless-stopped` and a host-mounted `./data` directory for the SQLite database.
- Existing VM containers, networks, Nginx sites, and certificates are not modified.

## Error handling and safety

- Invalid organiser credentials return `401` with `Invalid username or password`.
- Missing organiser session returns `401`; disallowed organiser routes return `403` where appropriate.
- Invalid guest RSVP token returns `404`, without identifying whether a guest exists.
- API validation returns field-specific `422` responses.
- Every route returns JSON; no HTML error pages are generated by the application.
- CORS permits only the Vercel frontend origin and supports credentialed requests.

## Verification

- Unit-test authentication, invitation expansion for `all_events`, token isolation, task mutation, and activity writes.
- Exercise the container locally with a persisted SQLite volume.
- Before VM deployment, confirm port `8040` is unbound and `nginx -t` succeeds.
- After deployment, confirm HTTPS redirects, certificate validity, health endpoint response, and browser credentialed requests from `wedding.skdev.one`.
